Where Does Normalisation Fit?

In previous weeks you built a conceptual model (an ER diagram) and then transformed it into a logical model (tables with primary keys and foreign keys). Normalisation is a technique for checking whether those tables are well designed. If they have problems, normalisation gives you a structured method for fixing them by splitting tables apart.

Figure 1: Normalisation sits between logical design and physical implementation.

Normalisation can also be used on its own as a "bottom-up" design technique, where you start from raw data (such as a form or spreadsheet) and work towards well-structured tables. In this module, you will mostly use it as a validation step after creating your logical model.

Reference: The textbook (Section 14.2) describes both approaches. For the coursework, think of normalisation as quality control for your tables.

The Problem: Data Redundancy

Imagine you store staff and branch information together in a single table. Every row for a staff member also includes the full address of the branch they work at. If three people work at Branch B1, the address 42 Victoria Rd appears three times. This repeated information is called data redundancy.

StaffIdNameStaff AddressBranchBranch Address
S1Fred Bloggs23 Acacia GardensB142 Victoria Rd
S2Bill Sykes17 Mafeking TerraceB142 Victoria Rd
S3George Shaw42 Privet DriveB142 Victoria Rd
S4John Doe5 Mornington CrescentB2112 King St
S5Tommy Atkins10 Rillington PlaceB2112 King St

Redundancy wastes storage space and makes data entry slower. More importantly, it creates the conditions for three types of error called anomalies.

3

The Three Anomalies

An anomaly is a situation where data becomes inconsistent, or where you lose information that you did not intend to lose. Anomalies happen because a table is trying to store information about more than one topic at the same time.

Insertion Anomaly

You cannot add a new branch until it has staff, because the primary key (StaffId) cannot be NULL.

Update Anomaly

Changing a branch address means updating every row for every staff member at that branch. Miss one and the data is inconsistent.

Deletion Anomaly

If the last staff member at a branch leaves and you delete their row, you also lose the branch address.

The fix is to separate the data into two tables: a Staff table and a Branch table. The branch address is stored once in the Branch table, and the Staff table uses a foreign key (Branch#) to link to it. This removes the redundancy, and all three anomalies disappear.

Figure 2: Splitting the combined table removes redundancy and prevents all three anomalies.
Careful: Some redundancy is expected and necessary. Foreign keys are copies of primary keys, and they need to be repeated in related tables. The redundancy we want to eliminate is non-key data being repeated, such as branch addresses appearing in every staff row.
4

First Normal Form (1NF)

A table is in First Normal Form (1NF) if every cell contains exactly one value. Two things violate this rule: multi-valued attributes (several values crammed into one cell) and repeating groups (a set of columns that repeat for multiple entries).

Violates 1NF ✗

Department D001 has "x2745, x2746" in the PhoneNumbers column. That is two values in one cell.

Fixed for 1NF ✓

Create a separate DeptPhone table. Each row holds one department ID and one phone number.

How to fix multi-valued attributes: move the multi-valued data into a new table, keeping a copy of the original primary key as a foreign key. For the Department/Phone example, you end up with a Department table (DeptId, DName) and a DeptPhone table (DeptId, PhoneNumber), where the primary key of DeptPhone is the combination of both columns.

Repeating groups work the same way. In the lecture slides, an order can contain many parts. The part details (PartNo, Description, UnitPrice, Quantity) repeat for each order. You fix this by pulling the repeating group into its own table, with a copy of the original key (OrderNo) included. The new table then has a composite primary key made up of OrderNo and PartNo together.

Figure 3: Removing repeating groups. Each cell now has exactly one value. The new table has a composite PK (OrderNo + PartNo).
Notice: The composite key from this step becomes important in Section 6 (2NF). Partial dependencies can only exist when the primary key has more than one column.
5

Functional Dependency

Before you can understand 2NF and 3NF, you need to understand functional dependency. This is the core concept behind the entire normalisation process.

A functional dependency is a rule that says: "if you know the value of column A, you can look up exactly one value of column B." We write this as A → B, and read it as "A determines B" or "B is functionally dependent on A." The column on the left side of the arrow is called the determinant.

Figure 4: Functional dependency works in one direction. StudentID determines surname, but surname does not determine StudentID.

A primary key, by definition, determines every other column in its table. That is actually what makes it a valid primary key: if you know the key, you can find every other value in that row.

When the primary key is a composite key (made up of two or more columns), we also need to think about full functional dependency. This means that a non-key column depends on all of the key columns together, and would stop working if you removed any of them. For example, in an OrderItem table with the composite key (OrderNo, PartNo), the Quantity depends on both columns together. You need to know which order and which part to look up how many were ordered.

Example: In the Purchase_Item table, OrderNo, PartNo → Quantity is a full functional dependency, because you need both OrderNo and PartNo to determine the quantity. However, PartNo → Description is only a partial dependency on the composite key, because you only need PartNo to know the description. The OrderNo is irrelevant; a bolt is a bolt regardless of which order it appears in.
6

Second Normal Form (2NF)

A table is in Second Normal Form (2NF) if it is already in 1NF, and every non-key column is fully functionally dependent on the entire primary key. In other words, there are no partial dependencies.

Notice: If a table's primary key is a single column (not composite), the table is automatically in 2NF. Partial dependencies can only exist when the key has two or more columns, because "partial" means "depends on part of the key." A single-column key has no parts.

Look at the Purchase_Item table from the lecture. The primary key is (OrderNo, PartNo). Quantity and ItemPrice depend on both columns together, so they are fine. But Description and UnitPrice depend only on PartNo. They are partially dependent on the composite key.

Figure 5: Splitting the table to remove partial dependencies. Descr and UnitPrice now live with their true determinant (PartNo).

How to fix it: move the partially dependent columns into a new table, along with the part of the key they actually depend on. That part of the key becomes the primary key of the new table. It also stays in the original table as a foreign key, keeping the link between the two.

Careful: When you split a table, the determinant appears in both tables. In the new table it becomes the primary key. In the original table it stays as a foreign key. Do not remove it from the original; otherwise, you lose the connection between the tables.
7

Third Normal Form (3NF)

A table is in Third Normal Form (3NF) if it is already in 2NF, and no non-key column depends on another non-key column. Every non-key column must depend directly on the primary key. When a non-key column depends on the primary key only indirectly, through another non-key column, this is called a transitive dependency.

Consider the Purchase Order table from the lecture. The primary key is OrderNo. OrderNo determines SuppNo (each order comes from one supplier), and SuppNo determines SuppName and SuppAddress (each supplier has one name and address). So we have a chain:

Figure 6: A transitive dependency. SuppName depends on OrderNo, but only indirectly, through SuppNo.

This causes the same redundancy problem again. Supplier details get repeated for every order from that supplier. If the supplier changes address, you have to update every order row.

How to fix it: create a new Supplier table with SuppNo as its primary key, containing SuppName and SuppAddress. Leave SuppNo in the Order table as a foreign key. Now each supplier's details are stored once.

Example: After the split, the Order table becomes Order(OrderNo, SuppNo*, DelDate, OrderDate, TotalPrice) and a new table Supplier(SuppNo, SuppName, SuppAddress) holds the supplier details. The asterisk on SuppNo marks it as a foreign key.

Putting It All Together

Normalisation is a series of tests applied in order. Each step checks for one specific type of problem and fixes it by splitting a table. Here is the full process:

Figure 7: The normalisation process from UNF to 3NF.

Sometimes a table is already well designed and does not need any changes. The lecture shows a StudentModule table with a composite key (StudentID, ModuleID) where CWGrade and ExamGrade are fully functionally dependent on the whole key, and there are no transitive dependencies. That table passes all three tests without any splitting. This is a perfectly valid outcome.

Good to know: If you designed your tables carefully from an ER diagram in week 3, normalisation may simply confirm that your design is correct. The ER-to-logical transformation rules naturally avoid many of the problems that normalisation catches.

How this connects to the lab: tutorial question 6 asks you to normalise the Spo-Tiff-Aye data from 0NF to 3NF. The lab then has you build that same database in SQL Server, creating tables with primary keys and adding foreign keys using ALTER TABLE statements. The tutorial is the "why" (designing the tables through normalisation), and the lab is the "how" (implementing them in SQL).

Careful: The lab mentions that the Features and Follow tables "are not in the ER Diagram." This is because those tables are linking tables that emerge during logical design. In the ER diagram, "a playlist contains many tracks" is shown as a line between Playlist and Track. The Features table is that many-to-many relationship, implemented as a table with foreign keys pointing to both Playlist and Track. Similarly, Follow is the linking table for the self-referencing User-follows-User relationship. These tables represent relationships, which is why they do not appear as separate entities in the conceptual model.
